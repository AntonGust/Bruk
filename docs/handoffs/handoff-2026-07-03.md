## What Phase 3.1 established (operational contract)

- **Initdata is declarative:** workload manifests carry `${INITDATA_B64}`; the blob lives in
  `gitops/apps/cluster.yaml` (`postBuild.substitute`). Regenerate with `build-initdata.sh` when
  `initdata.toml` changes. **Changing the blob rolls BOTH CC pods** (24B re-downloads ~90 GB,
  ~30 min).
- **Helm values are cluster changes now:** `install/helm-values/*.yaml` flow through a
  configMapGenerator into the HelmReleases — helm-controller upgrades on edit. ⚠️ In
  `gpu-operator.yaml`, `ccManager.defaultMode` is live GPU state: committing `"off"` force-flips
  CC mode node-wide (the ~19-min-outage class). The file now correctly says `"on"`.
- **Adoption protocol (reusable for any future Helm adoption):** `helm get values` diff →
  fix committed values to match live → rendered-manifest diff (`helm template` vs
  `helm get manifest`; expect only `helm.sh/hook` artifacts + regenerated Hubble TLS certs) →
  adopt → verify zero pod churn.
- **bootstrap.sh final shape:** host preflight → k3s → Helm ×3 (FIRST install only — Cilium is
  the CNI, Flux can't schedule without it) → wait for CC node → `flux install` (pinned v2.9.0) +
  deploy-key check + apply `gitops/gotk-sync.yaml` → wait for convergence. Everything else is Git.

## Product-architecture direction settled in discussion (record for Phase 3.2)

Q&A during the session pinned down how the customer surface should work — these shape the
operator design:

- **Customers never touch cluster/YAML/host.** Surface = CRDs (`BrukTenant`, `BrukModel`,
  `InferenceService`) written by the portal/fleet plane into the tenant's Git; Flux applies;
  the operator reconciles. Portal writes, cluster pulls (read-only deploy key), operator
  converges.
- **CC on/off is NOT a per-request toggle** (node-level property; mixed state is the outage
  trap). Product shape: provisioning-time tier ("confidential cluster" vs "standard"), or an
  operator-choreographed transition (drain GPU pods → flip → restore) as a deliberate,
  downtime-communicated operation.
- **"Install packages" = bring-your-own OCI image** (signed, digest-pinned, seeded into the
  tenant mirror, referenced from the CR). Never host access; never mutable containers.
  Whether tenants also get a namespace-scoped kubeconfig for auxiliary pods = open product
  decision (lean: CRD-only for pilot; needs an ADR when it matters).
- **Tenant repos REFERENCE the platform, never fork it:** platform repo ships versioned
  releases (`bruk-vX.Y.Z`, ideally as OCI artifacts); a tenant repo is thin — version pin +
  per-cluster substitutions + the customer's CRs. Upgrades = bump one pin (canary-able).
  NOTE: today `AntonGust/Bruk` plays both roles (platform + tenant-config for `anton-bruk` —
  the hardcoded blob/LV names in `gitops/` are really tenant config). Split when the fleet
  plane materializes; fine as-is for one cluster.
