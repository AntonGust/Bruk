## What Phase 2 established (the 30-second version)

Two pinned-stack mechanisms (kata 3.29.0 / guest-components `de3f6ff`), both = in-guest LUKS2,
ephemeral key, host sees ciphertext:
1. **Weights → plain emptyDir** (`emptydir_mode = "block-encrypted"`, RFC #247, default-on).
   First-run HF download (~27–33 min cold start); survives container restarts, dies with the pod.
2. **Image → `/dev/trusted_store`** (RFC #123): volumeMode:Block PVC at that magic devicePath →
   agent mounts LUKS2 over `/run/kata-containers/image` *before* guest-pull. Guest RAM: 24B
   160Gi→64Gi, smoke →32Gi.

⚠️ **The gotcha that cost half a day:** image-rs's default 3-way-parallel layer unpack fails
against the trusted store ("Failed to unpack layer to destination", ~11 s in, any RAM size,
mirror or docker.io). **Fix: `max_concurrent_layer_downloads_per_image = 1`** in initdata
`[image]` (committed in `manifests/registry/initdata.toml`). Candidate upstream bug — repro
trail in `docs/h100-bringup-status.md`. Also: CC guests resolve IPv6-first with no v6 route →
`gai.conf` ConfigMap (in the 24B manifest) fixes HF egress; kata debug console does NOT work on
these TEE guests (vsock timeout) — bisect with probe pods instead; **one pod per trusted-store
PVC** (two would both LUKS-format it).

## Standing gotchas (unchanged)

- Never mix CC + non-CC GPUs on the node; clean CC flips only with ALL GPU pods scaled to 0
  (~1 min/direction when clean; ~19 min outage when not).
- Reboots take 20–30+ min (PSP RMP init over 501 GiB) — not a hang.
- `kubectl logs` is empty for CC pods — use `exec`, `lastState`, host `journalctl`.
- Models >~70B-FP8 exceed one H100 (94 GB) → multi-GPU CC (PPCIE) waits for HGX B300 (Oct 2026).
