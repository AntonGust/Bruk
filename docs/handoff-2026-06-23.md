## What we accomplished today (in order)

1. **Drafted and pinned the Day-4 manifest** (`manifests/day4-vllm.yaml`): `vllm/vllm-openai:v0.11.1`
   (same vLLM version as NVIDIA's CUDA-13.1 release 25.12 — matches the box's in-guest CUDA 13.1),
   FP8, `max-model-len=32768`, `gpu-memory-utilization=0.90`, 32 Gi guest memory (Kata sizes the VM
   from pod limits; the 8 Gi default is too small for a 24B load), `/dev/shm` emptyDir.
2. **Created the gated-model secret** (`hf-token`) on the box.
3. **Three failures, all fixed — none GPU-related:**
   - `--attention-backend=flash_attn` → `unrecognized arguments`. It's **not a serve CLI flag**;
     moved to env `VLLM_ATTENTION_BACKEND=FLASH_ATTN` (FlashInfer JIT is broken on sm_120 with
     CUDA-13 wheels, so FLASH_ATTN is the working backend).
   - Engine core crashed: `ValueError: VLLM_PORT 'tcp://10.43.x.x:8000' appears to be a URI`. The
     **Service was named `vllm`**, so k8s injected `VLLM_PORT=tcp://…`, which vLLM mis-reads as its
     own config (expects an int). Renamed the Service to **`vllm-svc`**.
   - `kubectl logs` looked frozen at `Starting to load model` for ~15 min — actually the **silent
     HF weight download** (no TTY → no progress bars). Confirmed live via
     `du -sh /root/.cache/huggingface` climbing 41 → 73 → 90 GB. The repo ships **both** consolidated
     and sharded weights, so the pull is ~90 GB (≈2× the bf16 size).
4. **Checkpoint 4 verified.** After load + FP8 quantize + CUDA-graph capture the pod went `1/1 Ready`;
   `curl /v1/chat/completions` (run from inside the pod) returned a coherent answer. Timed 256-token
   generation = **~33.8 tok/s** single-stream.
