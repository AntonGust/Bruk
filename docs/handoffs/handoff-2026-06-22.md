## The blocker — diagnosed and fixed

### What it was NOT
The 06-18 large-BAR/MMIO-aperture theory was **wrong**. We injected
`-global q35-pcihost.pci-hole64-size=256G`, confirmed via an argv-logging wrapper that it reached
QEMU, and it **still segfaulted at the identical code offset**. Aperture sizing was not the cause.

### The real root cause (found via gdb + strace)
- gdb backtrace: `error_vprepend (util/error.c) ← vfio_pci_realize (hw/vfio/pci.c:3553) ←
  pci_qdev_realize`. The failing call is `iommufd_cdev_attach()` returning `false` into
  `vfio_pci_realize` (pci.c:3448) **without setting an Error**; QEMU then calls `error_prepend` on a
  NULL error → **NULL deref → SIGSEGV** (which masked the real failure as a bare "segfault").
- strace: `ioctl(/dev/iommu, _IOC(_,0x3b,0x92,_)) = -1 ENOTTY`. `0x3b`=`;`=IOMMUFD, `0x92` =
  **`IOMMUFD_CMD_VDEVICE_ALLOC`**. `ENOTTY` = the kernel's iommufd doesn't implement it.
- `IOMMU_VDEVICE_ALLOC` merged in **Linux 6.13** (iommufd vIOMMU Part-2 / vDEVICE). Host was **6.8**.
- QEMU 10.2 always associates a vIOMMU/vdevice for iommufd VFIO devices ("Replace iommu_device by
  vIOMMU"), so 10.2 specifically requires kernel ≥6.13. This is **GPU-size-independent** — any GPU on
  this Kata/QEMU stack + a <6.13 kernel hits it.

### The fix (verified)
```bash
sudo apt update && sudo apt install --install-recommends linux-generic-hwe-24.04   # -> 6.17.0-35
sudo reboot
```
Staged/validated before reboot: 6.17 initramfs carries `/etc/modprobe.d/vfio.conf` (vfio binding) +
nouveau blacklist; GRUB default + `iommu=pt` correct. Used a one-time-boot safety net (6.8 fallback)
for the first reboot, then promoted 6.17 to permanent default. Reboot took ~9 min (big-RAM POST).
After reboot: all 4 GPUs rebound to vfio-pci, cluster healthy, **0 segfaults**.

## SEV-SNP / fTPM (confidential-compute milestone — not blocking the vLLM spike)

State on 6.17: firmware already reserves the **RMP table** and the IOMMU advertises SNP, base SEV is
on, but `kvm_amd: SEV-ES/SEV-SNP disabled (ASIDs 0-0)`, `sev_snp=N`, and `/dev/tpm*` absent.

- **`iommu=pt` REMOVED (host-side, done):** SNP can't init in IOMMU passthrough mode
  (`AMD-Vi: SNP: IOMMU ... cannot be supported`). Dropped `iommu=pt` from `GRUB_CMDLINE_LINUX` →
  that error cleared, and GPU passthrough still works (re-verified). Backup: `/etc/default/grub.bak-iommupt-*`.
  `iommu=pt` was never needed for VFIO.
- **Remaining BMC/CBS pass:**
  - `AMD CBS → CPU Common Options → SEV-ES ASID Space Limit` — currently **1** (→ zero ES/SNP ASIDs).
    Set to **~100** (gives 99 SEV-ES/SNP-capable VMs; pool is 1–1006). Set "Control = Manual" if present.
  - `SEV-SNP Support = Enabled`; confirm `SMEE = Enabled`, `SNP Memory (RMP) Coverage = Enabled` (already on).
  - **fTPM:** disabling SPI/dTPM did **not** bring up fTPM — still no TPM. Need a *positive* enable:
    **Security Device Support = Enabled** + **TPM Device Selection = AMD CPU fTPM** (often outside AMD
    CBS, in Trusted Computing/Security; or AMD CBS → FCH Common Options). PSP is up, so fTPM is capable.
- **Verify after the pass + reboot:**
  ```bash
  dmesg | grep kvm_amd                            # want: SEV-SNP enabled (ASIDs 1 - 99)
  cat /sys/module/kvm_amd/parameters/sev_snp      # want: Y
  ls /dev/tpm0 ; dmesg | grep -i tpm              # want: tpm0 present, TPM 2.0
  ```
