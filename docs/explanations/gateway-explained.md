# The Gateway, Explained

This explains the Gateway design from ADR-0010 — first in simple terms, then every component in
detail. Read the first half to understand *what* the Gateway is and why it looks the way it does;
read the second half when you need to understand *how* each piece works.

---

## Part 1 — The Simple Version

### The problem

Today the models on the box serve tokens, but only from *inside* the cluster. There is no door to
the outside world — that was a deliberate security property (ADR-0008: "ClusterIP-internal
only"). A customer with an OpenAI SDK has nothing to point it at.

The Gateway is that door. Exactly one door, heavily guarded.

### The one-paragraph version

The Gateway is a single Envoy proxy pod, placed in front of all the model services. It listens on
the node's public port 443. Every request must show a valid API key or it gets a 401. Valid
requests are inspected — the `"model"` field in the request body says which model the customer
wants — and forwarded to the matching internal vLLM service. The customer never learns anything
about what's behind the door: to them, it looks exactly like the OpenAI API.

### The front-desk analogy

Think of the cluster as a private research building:

- **The Gateway is the front desk.** It's the only entrance; every other door is welded shut.
- **TLS is the soundproof booth** — nobody outside can eavesdrop on your conversation with the
  desk.
- **The API key is your access badge.** No badge, no conversation — the desk won't even confirm
  which labs exist (auth happens *before* routing, so an invalid badge always gets the same
  answer).
- **The receptionist reads your request slip** (the `"model"` field in the body) and walks it to
  the right lab (the InferenceService).
- **The visitor directory** (`GET /v1/models`) lists which labs are open right now — and the desk
  and the directory are printed from the same list, so they can never disagree.
- **The bouncer's counting clicker** (rate limiting) makes sure no single visitor — or crowd —
  can flood the building, badge or not.

### The request's journey, end to end

```text
Customer's OpenAI SDK
    │  https://<hostname>/v1/chat/completions   ("model": "mistral-24b", Bearer key)
    ▼
Node public IP, port 443  (Envoy binds it directly — hostNetwork)
    ▼
[1] TLS terminated           (cert from Let's Encrypt, auto-renewed)
    ▼
[2] Rate limit check         (token bucket — is the building too busy?)
    ▼
[3] API key check            (sidecar compares the Bearer key — 401 if wrong)
    ▼
[4] Body routing             (read "model" field → pick the upstream)
    ▼
[5] Forward to InferenceService's internal ClusterIP Service
    ▼
vLLM (inside the confidential guest) generates tokens
    ▼
[6] Response streams back through Envoy, unbuffered (SSE)
```

### The three design choices that shaped everything

1. **The public URL shape is forever.** Once a customer's code points at the endpoint, changing
   the URL scheme breaks them. So the *shape* ships correct on day one (one base URL, model in
   the body, `/v1/models`), even though only two models sit behind it.
2. **The operator renders everything; nothing configures itself.** The Gateway's whole
   configuration — routes, catalog, rate limits — is generated by the Airon operator from the
   CRs, exactly like the vLLM workloads. Same golden tests, same Git audit trail, no second
   control plane.
3. **The operator must never see a secret.** API keys and TLS private keys flow from Kubernetes
   Secrets straight into the Gateway pod as mounted files. The operator writes config that
   *points at* those files but can never read their contents. This one constraint decided the
   entire auth design.

## Part 2 — Every Component in Detail

### 2.1 Envoy itself — and why *plain* Envoy

Envoy is a high-performance L7 proxy (originally from Lyft, now CNCF-graduated; it's the data
plane inside Istio, Cilium's ingress, and most service meshes). Its configuration model has four
core nouns worth knowing because they appear everywhere below:

- **Listener** — a port Envoy accepts connections on (here: one HTTPS listener on :443).
- **Filter chain** — the ordered pipeline every request passes through (TLS → rate limit →
  ext_authz → Lua routing → router).
- **Route** — a rule mapping a request to a destination (here: mostly derived from the `"model"`
  header the Lua filter sets).
- **Cluster** — a named upstream destination (one per Ready InferenceService, pointing at its
  ClusterIP Service DNS name).

Three ways of running Envoy were considered:

| Option | Why not |
|---|---|
| Cilium's built-in ingress/Gateway API | Standard `HTTPRoute` cannot route on a JSON body field, so we'd be hand-feeding raw Envoy config through `CiliumEnvoyConfig` anyway — plus enabling it is a Helm change to the live CNI on the single node (the blast-radius trap). |
| Envoy Gateway (upstream control plane) | A second controller with its own reconcile loop, whose Gateway API abstractions the body-routing requirement immediately punctures. Earns its keep at many-routes/many-listeners scale we don't have. |
| **Plain Envoy, operator-rendered** ✅ | A Deployment + ConfigMap + Service produced by `operator/internal/render`, golden-tested like the workloads. Full config control (body routing is just config), zero new controllers, zero CNI changes. |

The trade-off accepted: we own the Envoy config and its upgrades ourselves. Since the operator
*is* the product's control plane, that's the point rather than a cost.

### 2.2 Where the configuration lives: `BrukTenant.spec.gateway`

The Gateway is per-cluster platform config, so it lives on the cluster contract object
(ADR-0007's definition of `BrukTenant`), as a new **optional** field — additive to v1alpha1, no
API version bump:

```text
BrukTenant "cluster"
└── spec.gateway              (absent = no public endpoint, today's behavior)
    ├── hostname              ← delivered via ${GATEWAY_HOSTNAME} postBuild.substitute
    ├── tls.secretRef         ← a standard kubernetes.io/tls Secret
    └── apiKeysSecretRef      ← Secret with named key entries
```

The hostname arrives through Flux's `postBuild.substitute` (the same pipe `${INITDATA_B64}`
uses) so the public repo never carries the reference box's coordinates.

**What makes a model routable: nothing — it's automatic.** Every Ready InferenceService is served
under its `servedModelName`. There is no opt-in or opt-out flag. This is deliberate: ADR-0007
defined the catalog as "a read-only join of BrukModel specs with Ready InferenceService
statuses," and the routing table is *the same join*, computed in the same render pass — so the
directory and the doors can never disagree. Consequence: the smoke model (0.5B) is publicly
listed and invocable behind auth. That's treated as a feature — a cheap, always-on, end-to-end
health check of the entire public path.

### 2.3 TLS: termination, SDS, cert-manager, DNS-01

**Termination.** Envoy terminates HTTPS at the listener. Traffic to the upstream vLLM services is
plain HTTP inside the cluster (the confidentiality story for data-in-use is the TEE, not the hop
between Envoy and the guest — and that hop never leaves the node today).

**How Envoy gets the cert — filesystem SDS.** SDS (Secret Discovery Service) is Envoy's mechanism
for loading TLS material dynamically. In filesystem mode, Envoy watches the mounted Secret files
and picks up a renewed certificate *without a restart or dropped connection*. The
`kubernetes.io/tls` Secret is volume-mounted into the pod; kubelet refreshes the mount when the
Secret changes; Envoy notices and rotates. Nobody restarts anything.

**How the cert gets into the Secret — cert-manager.** cert-manager is a standard Kubernetes
controller that speaks the ACME protocol to Let's Encrypt. It ships as one more Flux-managed
HelmRelease (the exact M3 adoption pattern). It proves domain ownership via **DNS-01**: it
creates a temporary TXT record under airon.ai, Let's Encrypt checks it, and the cert is issued.

**Why DNS-01 and not HTTP-01.** HTTP-01 requires answering a challenge on port 80 — awkward when
Envoy owns :443 in host network mode, and it would mean opening a second public port forever.
DNS-01 needs *no* open ports: issuance is completely independent of Envoy's bind. Since airon.ai's
nameservers are Cloudflare, cert-manager's native Cloudflare solver works with an API token —
scoped to the airon.ai zone only, stored as a Secret that **only cert-manager** can read (it
never touches the operator or the Gateway pod).

```text
cert-manager ──(Cloudflare API token)──► TXT record ──► Let's Encrypt validates
     │
     └──► writes kubernetes.io/tls Secret ──► mounted into Gateway pod ──► Envoy SDS hot-rotates
```

The producer/consumer split is deliberate: the Gateway just consumes whatever the Secret holds.
A future cluster could produce its cert differently (different DNS host, internal CA for a
private deployment) with zero schema or Gateway changes.

### 2.4 Authentication: Bearer keys and the ext_authz sidecar

**The scheme.** `Authorization: Bearer <key>` — chosen because every OpenAI SDK sends exactly
this header without any configuration. Any other scheme breaks "point your existing client at
us."

**Key storage.** One Secret, multiple named entries (`key-primary`, `key-secondary`, …). Rotation
is overlap-then-retire: add the new key, migrate clients, delete the old one. No hard cutover.

**The constraint that shaped the design.** ADR-0008 hardened the operator so it can `get` Secrets
but never materializes their values — verified by leak-guard regression tests. So the operator
*cannot* write API keys into the Envoy config it renders. Whatever validates keys must read them
through some path that bypasses the operator entirely.

**The mechanism: ext_authz.** Envoy's external-authorization filter pauses each request and asks
a separate service "allow or deny?" That service here is a **sidecar container in the same pod**
— and it is literally the `bruk-operator` binary started with a second entrypoint
(`gateway-authz`). One image, one digest pin, one CI pipeline, the operator's existing TDD and
review discipline — for what is security-critical code.

```text
        Gateway pod (hostNetwork)
┌──────────────────────────────────────────┐
│  Envoy ──ext_authz (localhost)──► authz  │
│    │                              sidecar│
│    │ mounted files:                 │    │
│    │  - TLS Secret (SDS)            │    │
│    │  - xDS config (ConfigMap)      │    │
│    │                    api-keys Secret  │
│    │                    (mounted file)   │
└────┼─────────────────────────────────────┘
     ▼ upstream vLLM ClusterIP Services
```

The sidecar volume-mounts the API-key Secret, compares the presented key using a constant-time
comparison (`crypto/subtle` — an ordinary `==` leaks timing information), and re-reads the file
when kubelet refreshes the mount, so key rotation needs no restarts either.

**Why not the two lighter options:**

- *Lua filter comparing keys*: the comparison would be hand-rolled security code inside an
  embedded scripting sandbox — no unit tests, no constant-time primitives, and every future
  security review has to reason about a Lua snippet inside a ConfigMap.
- *Envoy's native `api_key_auth` filter*: takes its credentials inline in the filter config —
  which is precisely what the operator is forbidden from rendering. (If a file-sourced credential
  mode verifies upstream someday, it's a noted simplification candidate.)

**Ordering matters.** Auth runs *before* routing. An invalid key gets a 401 regardless of whether
the requested model exists — the endpoint never leaks its catalog to unauthenticated callers.
After auth, a request for an unknown/not-Ready model gets an OpenAI-style `model_not_found` error
body, which OpenAI client libraries already know how to parse.

### 2.5 Model routing: reading the request body

The OpenAI contract puts the model name in the JSON **body** (`"model": "mistral-24b"`), not in
the URL. Standard HTTP routers route on path and headers, so this needs one extra step: a small
**Lua filter** (Envoy embeds LuaJIT for exactly this kind of glue) that parses the body's
`"model"` field and copies it into a request header; the route table then matches on that header
and picks the right upstream cluster.

The one rule that must never be violated: **buffer request bodies only, never responses.**
Reading the `"model"` field requires buffering the (small) request body — fine. But completions
stream back as Server-Sent Events over minutes; if Envoy buffered responses, the customer would
see nothing until the entire generation finished, destroying time-to-first-token. Related pinned
setting: Envoy's default route timeout is 15 seconds, which would kill any long generation —
route timeouts must be sized for multi-minute streams.

### 2.6 The catalog: `GET /v1/models`

The catalog response is **not served by a live service** — the operator renders the JSON during
reconcile (the join of BrukModel catalog metadata with Ready InferenceServices) and Envoy serves
it as static content via a `direct_response` route. Consequences:

- Catalog and routing table derive from the same computation in the same artifact — provably
  consistent, golden-testable.
- No extra runtime component that can be down, slow, or need RBAC on the CRs.
- Freshness equals reconcile-plus-propagation latency (about a minute) — fine for "a new model
  appeared."

The catalog sits behind the same Bearer auth as everything else (OpenAI requires auth on
`/v1/models` too, so compatible clients already send the key). The **only** unauthenticated path
on the entire Gateway is `/healthz` — a bare 200 for external monitoring.

### 2.7 Filesystem xDS: how config changes reach Envoy

xDS is Envoy's family of dynamic-configuration APIs — RDS delivers routes, CDS delivers clusters.
Usually a control-plane server streams these over gRPC. We use the simpler **filesystem
flavor**: the operator renders the RDS/CDS documents (and the catalog) into the ConfigMap, the
ConfigMap is mounted into the pod, and Envoy *watches the files*:

```text
InferenceService becomes Ready
    → operator reconciles: new cluster + route + catalog entry in ConfigMap
    → kubelet syncs the mounted files (~1 min)
    → Envoy detects the change, hot-swaps routes/clusters
    → model is publicly servable — zero pod restarts, zero dropped connections
```

Only the bootstrap config (listener, filter chain order, where to find the xDS files) is truly
static; changing *that* rolls the pod, but it changes about as often as the filter chain design
does. Everything that changes routinely — models appearing and disappearing — flows through the
watched files.

### 2.8 Exposure: hostNetwork on :443

The mechanics of getting public packets to the Envoy pod, given this cluster's constraints
(k3s installed with `--disable=servicelb`, Cilium at defaults, one node):

| Mechanism | Verdict |
|---|---|
| NodePort | Works, but confines to ports 30000–32767 → public URL grows `:30443`; some corporate egress blocks non-443. Extending the range = apiserver flag change. |
| hostPort | Requires Cilium portmap/kube-proxy-replacement features → live CNI Helm change. Ruled out. |
| MetalLB / Cilium LB-IPAM | Can't announce the node's own IP — needs its own routable VIP pool (DC IP allocation) plus L2/BGP announcement enabled in Cilium (live CNI change), all to route to the same single machine. Nothing to balance yet. |
| **hostNetwork: true** ✅ | Pod binds host :443 directly. Zero new components, zero CNI/k3s changes. |

hostNetwork means the pod shares the node's network namespace — a real widening of exposure for
an internet-facing pod, so it comes with the full hardening list as a **named, single-pod
exception**: `runAsNonRoot`, drop ALL capabilities except `NET_BIND_SERVICE` (the one capability
needed to bind a port below 1024 as non-root), read-only root filesystem, restricted seccomp
profile, no privileged mode, pinned/distroless Envoy image, NetworkPolicy where still meaningful.
Any future cluster policy that forbids hostNetwork must carry exactly one named exception: this
pod.

This is explicitly the **single-node-era mechanism**; the named successor is below.

One side effect worth knowing: with hostNetwork, "localhost" inside the pod *is* the host's
loopback — so the authz sidecar's port occupies host loopback and must not collide with anything
else on the node.

#### How this scales to multi-node and 500–1000 customers

The scale question has two different answers because of ADR-0001 (the cluster is the tenant
boundary):

**Fleet scale — 1000 customers means 1000 clusters, not 1000 tenants behind one Gateway.** Each
customer's cluster stamps out its own Gateway with its own hostname, keys, and cert; no Gateway
ever serves two customers. So the thing that must scale is the *pattern*, and it stamps cleanly.
A per-cluster Envoy is trivially sized: its traffic is bounded by what that cluster's GPUs can
generate — orders of magnitude below a single Envoy's capacity. The proxy is never the
bottleneck; the GPUs are.

The genuine fleet-scale pressure points are elsewhere: **DNS automation** (a record per cluster)
and **Let's Encrypt rate limits** — by default ~50 *new* certificates per registered domain per
week (renewals are exempt). That caps customer-onboarding pace, not steady state; an LE
rate-limit increase (routinely granted to providers) is needed before mass onboarding.

**Per-cluster scale — the multi-node successor.** When a cluster has multiple nodes (Spectrum-4
fabric, HGX B300), hostNetwork's weakness appears: the Gateway is pinned to one node's IP. The
named successor: the Gateway becomes a normal pod (1–2 replicas) behind a `LoadBalancer` Service
with a per-cluster VIP announced to the top-of-rack switches — via Cilium's BGP control plane or
a BlueField-3-hosted load balancer, decided when the fabric's actual routing design exists
(that's the one input missing today, which is why the ADR names the successor but doesn't
speculate on its details).

**Why switching later is cheap by design:** only the rendered exposure plumbing changes — the
Deployment stops being hostNetwork and gains a `LoadBalancer` Service. Hostname, `BrukTenant`
schema, TLS Secret, API keys, xDS mechanism, and the public contract are all untouched. Customers
see a DNS update at most.

### 2.9 Rate limiting: what the buckets protect

Envoy's **local rate limit** filter implements token buckets in-process — pure config, no new
components. Three layers, each protecting a different thing:

1. **A conservative global bucket on `/v1/*`** — protects the *engine queue*. One GPU saturates
   at low double-digit concurrent requests; the bucket's job is to keep vLLM's queue sane, not to
   implement per-customer QoS.
2. **A stricter bucket on unauthenticated requests** — makes credential-stuffing cheap to reject.
   An attacker probing keys burns their budget at the front desk, not in the auth path.
3. **Connection limits** — caps raw connection floods before they reach anything.

**Per-key quotas are deliberately deferred** to the metering increment (the same future work as
named-key audit logging in the sidecar): real per-customer QoS only means something when usage is
measured, and it needs a stateful global rate-limit service — a new component this phase doesn't
justify.

### 2.10 The secrets flow — who can read what

The design's quiet backbone. Three secrets, three different consumers, and the operator reads
none of them:

```text
API keys Secret ────────► authz sidecar only        (mounted file, constant-time compare)
TLS cert Secret ────────► Envoy only                (mounted file, SDS hot-rotation)
Cloudflare token Secret ► cert-manager only         (DNS-01 challenges)

Airon operator ─────────► renders config that POINTS at mounted paths;
                          never reads, logs, or embeds any secret value
```

If the operator is ever compromised, it still cannot exfiltrate a customer key or a TLS private
key — the ADR-0008 blast-radius bound survives this phase intact.

### 2.11 What this phase deliberately does not do

- **No per-key quotas, metering, or billing** — the metering increment.
- **No named-key audit logging** — same increment.
- **No Dynamo** — Envoy forwards straight to vLLM; when multi-node/Dynamo arrives (HGX B300,
  Oct 2026), it slots in *behind* the contract, invisibly to clients. This invisibility is the
  entire reason the Gateway owns the contract.
- **No multi-node load balancing yet** — hostNetwork is the single-node answer; the successor
  (per-cluster VIP via Cilium BGP or BlueField-3 LB) is named in §2.8 and gets decided when the
  fabric's routing design exists.
- **No customer-written CRs** — customers still never touch Kubernetes; a future portal/API
  writes CRs on their behalf (ADR-0008 non-goal, unchanged).
