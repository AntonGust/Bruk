# Bruk project plan — 2026-07

Sequencing, testing, and delivery plan for the next phase. Grounded in
`docs/handoffs/handoff-2026-07-02.md` (the three open workstreams), ADR-0006, and
`docs/deployment-model.md`. The short version: **prove it → scale it → productize it**
(Workstream B → A → C).

## Where we are (baseline, 2026-07-02)

Confidential serving is proven end-to-end for a small model (Qwen2.5-0.5B in a SEV-SNP Kata guest
with a CC-mode H100, ~378 tok/s warm, image via the local registry mirror), and the per-cluster
stack is reproducible from committed files (`install/` + `docs/RUNBOOK-confidential-serving.md`).
Everything is merged to `main`. Not done: the attestation is **unverified**, the real 24B model
doesn't fit the RAM-based storage path, and all automation above bootstrap scripts is designed but
not built.

---

## Phase 1 — Attestation verification (Workstream B) — *first, ~1–2 days*

**Why first:** smallest task, and it closes the core claim. Today we can say "it boots in SNP" but
not "it's cryptographically confidential." It also establishes the verification harness we re-run
after every storage/config change in Phase 2.

**Implement:**

1. **PSP report verification against AMD KDS** — `snpguest report` in a bare `kata-qemu-snp` pod
   (privileged; `mknod /dev/sev-guest c 10 257`), then `snpguest fetch ca/vcek genoa …`
   (EPYC 9224 = Genoa) and `snpguest verify certs` + `snpguest verify attestation`: full
   VCEK→ASK→ARK chain + report signature + our nonce. Needs guest egress to `kdsintf.amd.com`.
2. **GPU device attestation** — NVIDIA nvtrust local verifier inside the confidential guest
   (SPDM handshake + device-cert chain, via NRAS or local). Make the check explicit — a working
   vLLM is only partial evidence.
3. **Perf delta** — same small model (Qwen-0.5B) non-CC vs CC, warm ≥200-token requests; expect
   ~12–15 % overhead (Phala reference).

**Tested:** this phase *is* a test — pass criteria are cryptographic (chain validates, nonce
matches, nvtrust passes) plus a recorded perf number. The checks are packaged as repeatable
manifests + scripts (`manifests/attestation/`) so they become a regression suite, not a one-off.

**Delivered:** manifests/scripts committed; results + perf delta recorded in
`docs/h100-bringup-status.md`; ADR-0004 steps 4–5 closed; Sprint-2 teaching docs updated.
**Done when** an independent party could re-run the verification from the repo alone.

## Phase 2 — Block storage → confidential 24B (Workstream A / ADR-0006 Part 2) — *~1–2 weeks*

**Why second:** unblocks the actual product model (Mistral-Small-3.1-24B FP8). Highest-value
remaining pilot work, but bleeding-edge (CoCo RFC #123/#247 territory) — so it goes after the
cheap win, staged so each step is independently verifiable:

1. **Timeboxed research spike (1–2 days):** verify what kata 3.29.0 / guest-components `de3f6ff`
   actually support for image-store-on-block-device and attached encrypted volumes. Read the
   pinned source (the approach that cracked the mirror config). Decision gate: if trusted
   image-on-block isn't usable in this release, fall back to "weights on block device + small
   image via mirror" and note it in the ADR.
2. **Block device into the guest:** LVM LV on NVMe → virtio-blk → dm-crypt (or dm-verity
   read-only) in-guest. Smoke test: device visible and writable inside a bare SNP guest.
3. **Image cache off RAM:** point the guest image store at the block device. Smoke test with the
   *small* model first — verify via in-guest `df`/mount inspection that the unpack lands on NVMe,
   not tmpfs.
4. **Weights delivery:** decide pre-staged ciphertext vs first-run HF download into the volume
   (lean: first-run download first — simpler, persists — upgrade toward pre-staged/attested-key
   later). Record the decision in ADR-0006.
5. **The 24B itself:** deploy `manifests/h100-vllm-cc.yaml`, record tok/s.

**Tested:** each stage has an observable end-to-end check before the next; after the final config,
**re-run the Phase 1 attestation suite** (storage changes must not silently weaken the
confidential posture). Constraints honored: no CC/non-CC GPU mixing on the node; diagnose via
`lastState` + host journal (CC pod logs are empty).

**Delivered:** manifests + config committed; RUNBOOK gains a "large model" section; ADR-0006
amended with the weights decision. **Done when** the 24B serves confidentially with image+weights
on NVMe and the tok/s number is recorded.

## Phase 3 — Level-2 productization (Workstream C) — *the long track, staged*

Start once the pilot is "proven enough" (after M2). Three sequenced sub-phases, each
independently useful:

**3.1 GitOps (Flux)** — smallest, highest leverage. Wrap `install/helm-values/*` as HelmReleases
and `manifests/*` as Kustomizations; `bootstrap.sh` shrinks to "install Flux, point it at the
repo." *Tested:* convergence tests — delete a managed resource, watch it self-heal; ideally a
from-scratch rehearsal (a non-GPU VM cluster rehearses everything except the CC workload; the
reference box validates the GPU/CC layer). *Delivered:* the repo becomes the deployment artifact.

**3.2 Airon Operator (ADR-0002)** — Go + controller-runtime, CRDs (`BrukTenant`, `BrukModel`,
`InferenceService`), deterministic (no LLM in the loop). The project shifts from infra bring-up to
software engineering here, so conventional discipline kicks in: TDD (envtest unit tests +
kind-based integration tests, 80%+ coverage), GitHub Actions CI, code review before merge.
*Delivered:* versioned container image, deployed as a HelmRelease through the Flux repo from 3.1 —
the operator ships through its own GitOps pipeline.

**3.3 Fleet plane + bare-metal (ADR-0001)** — MAAS/PXE to image a Node and apply the host layer,
plus the cross-cluster pieces: tenant/key registry, host-attestation verifier, and eventually
KBS/Trustee for the Pattern-A upgrade (ADR-0006). Mechanism (decided 2026-07-07): **stock Ubuntu
24.04 + MAAS HWE kernel + cloud-init that *calls* the committed scripts at a pinned ref**
(`host-setup.sh`, then a post-reboot systemd oneshot for `bootstrap.sh`) — the scripts stay the
single source of truth; no golden image until drift/speed hurts (multi-node problem). Boot policy
(decided 2026-07-07): Nodes stay **disk-first**; MAAS forces PXE via the BMC (next-boot override)
only when re-provisioning — deliberate deviation from MAAS's PXE-first default, because this
firmware hangs on a dead PXE entry and Node reboots must not depend on the fleet-plane host.
Placement (decided 2026-07-07): the datacenter is Airon-owned. MAAS runs on its own dedicated
machine; a **separate fleet-plane host** carries the trust services (registry checkout + age key,
Keylime verifier/registrar, later KBS/Trustee). Imager and key-store/verifier are thus on
different machines — deliberate separation; keep it when adding sites.
*Tested:* acceptance = "provision a fresh node end-to-end with only BIOS done by hand."
Develop against a VM lab (same split as 3.1: VMs rehearse everything except BIOS/SEV-SNP/GPU);
final M5a acceptance is one **scheduled, sacrificial re-provision of the reference box** (decided
2026-07-07) — the real test of the "fresh clone reproduces the cluster" invariant. Time it for a
low-stakes window, only after the VM rehearsal converges unattended.
**Hard gate:** host attestation needs the discrete TPM (on order) — sequence the verifier after it
arrives; don't block the rest of 3.3 on it.

## Milestones

| Milestone | Meaning | Exit criterion |
|---|---|---|
| **M1 — Provably confidential** | Phase 1 done | Attestation validates against AMD/NVIDIA chains; perf delta recorded |
| **M2 — Product model confidential** | Phase 2 done | 24B serves from NVMe-backed confidential storage |
| **M3 — Declarative cluster** | Flux live | Fresh cluster converges from Git; self-heals |
| **M4 — Operator MVP** | CRD-driven serving | `InferenceService` CR → running confidential workload, CI green |
| **M5a — Fleet provisioning** | New node w/o hand-assembly | Imaging + host layer automated; only BIOS is manual per machine |
| **M5b — Attestation-gated onboarding** | Node trusted before secrets | A provisioned node receives secrets only via the attestation gate (TPM-backed host attestation); KBS/Trustee stays post-M5 (Pattern A, ADR-0006) |

## Cross-cutting: how everything is tested and delivered

- **Infra phases (1–2, 3.1, 3.3):** verification means *observable end-to-end behavior* —
  attestation chains validate, tokens are served, storage demonstrably lands on NVMe — and every
  check is committed as a re-runnable script/manifest. Doubles as regression suite and student
  material.
- **Software phase (3.2):** classic TDD + CI + review.
- **Delivery is the repo, at every phase.** Standing invariant: a fresh `git clone` + the runbook
  (later: + Flux) reproduces the cluster. Each milestone ends with a docs pass
  (RUNBOOK/status/ADR/teaching) and a session handoff.
- **Hardware timeline shapes the tail:** TPM arrival gates the host-attestation verifier; the
  Spectrum-4/BlueField fabric (summer 2026) and HGX B300 (Oct 2026) are when multi-node and
  multi-GPU-CC become real — keep 3.3 single-site until then, and recheck the storage story at
  B300 (already an ADR-0006 revisit trigger).
