## Workstream A — ADR-0006 Part 2: block-device storage for the 24B  *(recommended 2nd)*

**Goal:** serve the real **Mistral-Small-3.1-24B FP8** confidentially. **Why blocked:** confidential
containers guest-pull the image into a **RAM-backed tmpfs** (35 GB unpacked → OOM at any sane RAM), and
`shared_fs="none"` means the non-CC weights PVC can't mount. **The mirror (Part 1) is done** — the image
pulls fast, but still lands in RAM. Part 2 = get storage **off RAM onto NVMe** (3.2 TB free).

**The two sub-problems:**
1. **Image cache off RAM** — direct the guest image store to a block device (CoCo RFC #123 "Trusted
   Storage"). May be RFC/partly-implemented on kata 3.29.0 — **verify support first**.
2. **Weights delivery without virtiofs** — attach a block device (virtio-blk on NVMe) into the guest,
   encrypted (dm-crypt via `kata-agent`/CDH `luks-encrypt-storage`, or dm-verity read-only). CoCo RFC
   #247 "Trusted Ephemeral Data Storage" is exactly the "tmpfs wastes guest memory" fix.

**First steps:**
- Research the exact knob on kata 3.29.0 / guest-components `de3f6ff` for (a) image store on a block
  device and (b) an attached encrypted scratch/persistent volume. Check
  `configuration-qemu-nvidia-gpu-snp.toml` for a scratch-disk setting; check RFC #123/#247 status in
  this release. (Same "read the pinned source" approach that cracked the mirror config.)
- Provision an LVM LV / partition on the NVMe → attach as virtio-blk → dm-crypt inside guest → point
  image store + weights at it → deploy `manifests/h100-vllm-cc.yaml` (the 24B variant, kept for this).
- **Weights question to decide:** pre-stage the ~90 GB (host writes ciphertext, guest decrypts with an
  attested key — the Pattern-A/B "real" answer) vs. download from HF into the block device on first run
  (simpler; persists). Note both in the ADR when chosen.

**Files/refs:** `docs/adr/0006-confidential-weights-delivery-storage.md` (the decision), `manifests/
h100-vllm-cc.yaml` (the blocked 24B manifest), `docs/h100-bringup-status.md` (the failure detail).
**Gotchas:** don't try to re-enable virtiofs (`shared_fs=none` is a real QEMU snp-v3 bug, not just
policy); the block-device/image-store config may live in `cdh.toml` like the mirror did; `kubectl logs`
is empty for CC pods (diagnose via `lastState` + host `journalctl`).
**Done when:** the 24B serves confidentially with image+weights on NVMe (not RAM), tok/s recorded.

## Workstream B — Day-5 attestation verify  *(recommended 1st — smallest, closes the security story)*

**Goal:** prove the confidential path is *cryptographically real*, not just "boots in SNP", and record
the CC-vs-non-CC perf delta. This is ADR-0004 steps 4–5, still 🟠 EXPECTED. We pulled a PSP report
(Day 2) but never **verified** it; GPU attestation not run.

**First steps:**
- **PSP report → verify against AMD KDS:** in a bare `kata-qemu-snp` pod (privileged; `mknod
  /dev/sev-guest c 10 257`), `snpguest report` then `snpguest fetch ca/vcek genoa …` (EPYC 9224 = Genoa)
  and `snpguest verify certs` + `snpguest verify attestation` — show it *validates* (VCEK→ASK→ARK +
  report signature + your nonce), not just parses. Needs guest egress to `kdsintf.amd.com`.
- **GPU device attestation:** NVIDIA **nvtrust** (github.com/NVIDIA/nvtrust) local verifier inside the
  confidential guest — SPDM handshake + device-cert chain (via NRAS or local). Note the CC GPU may be
  locked (no CUDA) until attestation; a working vLLM is partial evidence — make the check explicit.
- **Perf delta:** run the **same** small model (Qwen-0.5B) non-CC vs CC, warm ≥200-token requests;
  expect ~12–15 % overhead (Phala reference). Record it.

**Files/refs:** `docs/teaching/sprint-2-instructor-answer-key.md` (Day-5 section has the tooling),
`docs/adr/0004` (steps 4–5). **Gotchas:** `snpguest` v0.10.0; `/dev/sev-guest` needs privileged +
mknod; guest needs internet to AMD KDS. **Done when:** PSP report validates against AMD's chain, GPU
attestation passes, perf delta recorded in `h100-bringup-status.md`.

## Workstream C — Level-2 GitOps / fleet automation  *(largest — the productization track)*

**Goal:** move from **Level-1** (bootstrap scripts) to **Level-2** — GitOps/Flux + the Airon Operator +
fleet-plane/MAAS provisioning — so a new site is "point Flux at the repo" and **only BIOS is manual per
machine**. See `docs/deployment-model.md`. None of this is built (repo = docs + manifests + install
scripts).

**Staged first steps:**
1. **GitOps (Flux):** install Flux, point it at the repo, wrap `install/helm-values/*` as HelmReleases +
   the `manifests/*` as Kustomizations so the per-cluster stack is declaratively reconciled/self-healing
   (retires `bootstrap.sh`). Smallest, highest-leverage first move.
2. **Airon Operator (ADR-0002):** scaffold the deterministic Go/controller-runtime operator + CRDs
   (`BrukTenant`, `BrukModel`, `InferenceService`) reconciling into the vLLM/registry/CC stack. Big.
3. **Fleet plane + bare-metal (ADR-0001):** MAAS (the box already has PXE/IPMI leftovers) to image a
   Node + apply the host layer (`host-setup.sh` logic via cloud-init); the cross-cluster control layer
   (tenant/key registry, host-attestation verifier — the latter needs the TPM).

**Files/refs:** `docs/deployment-model.md`, ADR-0001, ADR-0002, `install/` (the imperative recipe to
declarative-ize). **Gotchas:** BIOS stays manual (or Redfish/IPMI later); the operator must be
deterministic (no LLM in the control loop — ADR-0002); host attestation is gated on the missing TPM.
**Done when (phased):** Flux reconciles the per-cluster stack from Git; then the operator; then fleet
provisioning.

## Recommended order
**B → A → C.** B is small and *closes the confidential-serving story* (proves it's genuinely confidential
+ a real perf number). A unblocks the actual product model (24B), fragile but high-value. C is the
long platform track — start whenever the pilot is "proven enough" and productization is the priority.

## Standing gotchas (apply across all workstreams — learned the hard way)
- **Never run a CC GPU + non-CC GPU on this node at once**, and don't restart k3s/containerd while the
  operator's CC config disagrees with real GPU state — a `nvidia-cc-manager` restart force-resets GPUs
  node-wide (cost a ~19-min outage once). The node is all-CC now; keep it that way.
- **`cc_init_data` must be `base64(gzip(toml))`**; registry/image config lives in **`cdh.toml`**, not a
  standalone `registries.conf`.
- **`kubectl logs` is empty for CC pods** (guest stdout is in the TEE) — use `kubectl exec` + `lastState`.
- **Reboots = 20–30 min**; after any reboot re-check `cc.mode.state`/`cc.ready.state` before relying on CC.
