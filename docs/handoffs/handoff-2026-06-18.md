## THE BLOCKER — QEMU segfaults on Blackwell VFIO cold-plug

- A `kata-qemu-nvidia-gpu` pod hangs `ContainerCreating` → `FailedCreatePodSandBox: create
  container timeout`. Journal shows **`qemu-system-x86_64: segfault at 0`** immediately, before QMP
  capability negotiation. QEMU dies on startup.
- **Isolated:** plain `kata-qemu` (no GPU) boots fine → it is the **GPU VFIO cold-plug**, not Kata.
- **Root cause:** `lspci -v` on the GPU shows `Memory at 6a000000000 (64-bit, prefetchable)
  **[size=128G]**`. The 128 GB BAR overflows the guest's default 64-bit MMIO aperture; QEMU
  segfaults trying to map it. Known large-BAR datacenter-GPU passthrough issue.
- Component versions: QEMU **10.2.1 (kata-static)**, q35, `cold_plug_vfio = "root-port"`,
  `pcie_root_port = 8`, `default_memory = 8192`. Guest image is the NVIDIA-optimized
  `kata-ubuntu-noble-nvidia-gpu-590.48.01.image` + `vmlinuz-6.18.15-189-nvidia-gpu` (**driver baked
  into the guest — the in-guest-driver question is resolved**).

### Fix directions to try next (in order)
1. **Enlarge the guest 64-bit PCI hole / MMIO reservation** in
   `/opt/kata/share/defaults/kata-containers/configuration-qemu-nvidia-gpu.toml` (or via pod
   annotations). QEMU q35 needs a `pci-hole64`/`mem-reserve` large enough for a 128 GB BAR
   (e.g. machine `pci-hole64-size` / Kata `vfio` MMIO settings). This is the most likely single fix.
2. Confirm the host CPU `phys-addr-bits` and that the guest gets enough address space; ensure
   **Resizable BAR** isn't forcing the full 128 GB window when a smaller one would do.
3. Try a **cloud-hypervisor** runtimeClass variant (`kata-clh`) as a comparison — different MMIO
   handling; can confirm whether the issue is QEMU-specific.
4. As needed, capture the exact generated QEMU cmdline (enable kata debug logging) to see the
   VFIO/`-device vfio-pci` + machine args and the precise crash point.
