## Storage work performed

The docs expected a `bruk` VG on the spare Samsung NVMe. The device layout showed `/dev/nvme0n1` was
unused and `/dev/nvme1n1p2` held root.

Created:

```bash
sudo pvcreate /dev/nvme0n1
sudo vgcreate bruk /dev/nvme0n1
sudo lvcreate -L 120G -n trusted-image-smoke bruk
sudo lvcreate -L 150G -n trusted-image-24b bruk
```

Resulting devices:

- `/dev/bruk/trusted-image-smoke`
- `/dev/bruk/trusted-image-24b`

The smoke LV was later wiped after a failed secure-mount attempt:

```bash
sudo wipefs -a /dev/bruk/trusted-image-smoke
```

## Trusted-storage issue

The operator/trustedStore path still fails inside Kata/CDH:

```text
[CDH] [ERROR]: Secure Mount failed
```

Observed behavior:

- Volume mapping reached the Kata guest.
- The image was pulled/present.
- `/dev/bruk/trusted-image-smoke` showed a partial `DM_integrity` signature after the failed attempt.
- Wiping and retrying did not fix it.

Working conclusion: trusted-storage secure mount is unhealthy on this node/stack. Today's successful
local path intentionally bypasses `/dev/trusted_store` and uses a RAM-backed image-store fallback for
the small smoke workload.
