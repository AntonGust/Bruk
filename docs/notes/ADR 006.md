Good call to nail down the plan before building it. Here's the why and how, grounded in our specific stack (k3s + CoCo kata-qemu-nvidia-gpu-snp + confidential-data-hub/image-rs on the H100 box).

The two walls we're solving

We hit two independent failures getting a workload into the confidential guest, and each fix targets one:

├───────────────┼───────────────────────────────────────────────────────────────────────────────────────────────────┼────────────────────────────┤

Neither is a security-primitive problem (SNP + PSP attestation + CC GPU all work) — they're logistics of getting bytes into a TEE. That's exactly what ADR-0006 says to fix.

---
Part 1 — Local registry mirror

Why

A confidential guest guest-pulls its image: image-rs inside the encrypted guest fetches layers directly from the registry (bypassing the host's containerd cache — that's deliberate, so the untrusted host never mediates the image). On our box that in-guest pull to docker.io is failing, while the host's pull works — the guest's egress/DNS path for image-rs is flaky, and Docker Hub also rate-limits anonymous pulls (a real risk with N student teams each re-pulling 13 GiB).

A mirror on the local network turns a flaky, rate-limited, internet-dependent 13 GiB pull into a fast, reliable LAN fetch. And the error message itself — "from all mirror/mapping locations or original location" — tells us image-rs is built to consult a mirror mapping first. We're using a supported path, not hacking around one.

How

1. Run a registry on the host — registry:2 (or zot) as a container/pod, bound to the host IP, port 5000. ~10 lines.
2. Seed it from the image the host already has. The host already cached vllm/vllm-openai:v0.11.1, so skopeo copy docker://vllm/vllm-openai:v0.11.1 docker://<host-ip>:5000/vllm/vllm-openai:v0.11.1 — no re-download from docker.io.
3. Point the guest's image-rs at the mirror. This is the version-specific bit: CoCo image-rs reads a registry configuration (a mirror/mapping + optional auth) delivered to the guest. On our stack the likely delivery mechanisms are (a) the cc_init_data annotation (which I saw is enabled in configuration-qemu-nvidia-gpu-snp.toml's enable_annotations), or (b) a CDH/image-rs config resource. The mirror maps docker.io/vllm/vllm-openai → <host-ip>:5000/....
4. Point the manifest at the mirrored ref (or rely on the mapping so the manifest stays unchanged).

What to verify during build: the exact config surface image-rs reads on GPU-Operator v26.3.2 / kata-deploy 3.29.0 (mirror config format + whether it's cc_init_data, a registry.conf-style file, or a CDH resource). This is the one genuinely fiddly step.

Part 2 — Encrypted block device

Why

Everything the confidential guest writes currently goes to a RAM-backed tmpfs (SNP guest memory is private/encrypted, so there's no host-shared scratch). The 35 GB image unpacks there → OOM. And with shared_fs="none", there's no virtiofs to mount the weights PVC either. So both the image cache and the weights need somewhere that (a) isn't RAM, and (b) the host can't read.

An attached virtio-blk device on NVMe (we have 3.2 TB free) is that somewhere:
- Off RAM → the 35 GB image and tens of GB of weights fit trivially; no OOM, and weights persist instead of re-downloading every start.
- Encrypted (dm-crypt) with a guest-generated key → the host sees only ciphertext, so confidentiality is preserved. This is literally CoCo RFC #247 "Trusted Ephemeral Data Storage," which exists because "the tmpfs fallback wastes costly guest memory" — i.e. our exact OOM.

This is the Pattern B enabler: for read-only weights you can use dm-verity (integrity — host can't tamper) instead of/alongside dm-crypt, which is what Edgeless Continuum and Tinfoil actually ship.

How

1. Provision a backing device on the host — an LVM logical volume or a partition on the NVMe (dedicated, sized for image + weights, e.g. 200 GB).
2. Attach it to the guest as virtio-blk. Kata has a direct block-device assignment path; the device is presented to the guest, not host-shared.
3. The guest encrypts it. kata-agent + CDH run luks-encrypt-storage — dm-crypt with a random key generated inside the guest — so the host never holds the key. (This part is implemented in CoCo.)
4. Steer storage onto it:
  - Weights: mount the device as the model volume (replaces the impossible virtiofs PVC). For read-only weights, dm-verity for integrity.
  - Image cache: point the guest's image store (/run/kata-containers/image) at the device instead of tmpfs. ⚠️ This is the part that's partly RFC-stage (RFC #123 "Trusted Storage") — it may need a newer CoCo release or config spelunking. If it's not wired yet, we still win big by putting weights on the device and keeping only the image in RAM (a much smaller problem for a right-sized model).

What to verify during build: whether our CoCo/kata version supports directing image storage to the block device, or only ephemeral/weights volumes. That determines whether the image also comes off RAM or just the weights.

How they compose (the end state)

        ┌─────────────────────────── HOST (untrusted) ───────────────────────────┐
        │  registry:2  @ :5000        NVMe: /dev/…  (ciphertext only)             │
        │   (vllm image, seeded)          │  virtio-blk                           │
        └───────┬──────────────────────────┼────────────────────────────────────┘
                │ LAN pull (fast)           │ attached, dm-crypt (guest key)
        ┌───────▼───────────────────────────▼──────────── SEV-SNP guest (TEE) ───┐
        │  image-rs → mirror mapping        dm-crypt volume:                      │
        │  pulls image from :5000           • image cache (off RAM)               │
        │                                   • weights (persist, host can't read)  │
        │  vLLM loads weights → serves on the CC-mode H100                        │
        └────────────────────────────────────────────────────────────────────────┘

- Mirror kills the registry-egress wall and the re-pull cost.
- Block device kills the RAM-OOM wall and gives weights a home that shared_fs=none otherwise denied.
- Together they make a real-sized model servable confidentially — which is why the small-model brute-force (160 GB RAM) was only ever a proof-of-path, not the answer.

---
Honest scoping

- Mirror: low risk, high payoff. Standard registry + skopeo; the only unknown is the image-rs mirror-config format on our version. A few hours.
- Block device: medium risk. The encrypted ephemeral volume is implemented; steering image storage onto it is RFC-stage, so we may only get weights onto the device on our current CoCo version (still a big unblock). Half a day to a day of config work, possibly a CoCo version bump.
- Neither touches the proven security foundation (SNP guest, PSP attestation, CC GPU) — they're additive plumbing.

This is exactly the "platform sub-project, not a sprint task" line from ADR-0006 — which is why Sprint 2 targets a small model and leaves this as roadmap.
